# Hands-on change log (for Codex / next AI session)

Quick orientation for whoever picks this repo up next. This documents the
big frontend passes done in July 2026 on top of the Phase 1 mock-backend app
(customer, then barber, then shop owner). Read `ARCHITECTURE.md` and
`CODE-PATTERNS.md` first for the big picture; this file is only "what
changed recently and where."

Everything below can still run against the mock backend (`localStorage`, DB
version now **19**). The same UI can now use the Express-backed `ApiBackend` via
`VITE_DATA_BACKEND=api`; `DataBackend` remains the only contract the UI talks to.

## Supabase/Express migration progress

- The mock seed is credential-free at version 19. Historical bundled accounts
  and their dependent shop catalogue are removed while user-created accounts
  are preserved.
- Versioned Supabase schema, trigger, RLS, seed, and atomic API-write migrations
  live in `supabase/`.
- `apps/api` provides the protected Express REST layer with shared Zod
  validation and server-only service-role access.
- `ApiBackend` in `packages/shared/src/services.ts` persists the user session,
  refreshes expired access tokens, maps REST errors to `DataError`, and supplies
  chat subscriptions through authenticated polling.
- `apps/web/src/services/backend.tsx` switches between `mock` and `api` without
  component changes.
- The Docker-backed local Supabase stack passes direct RLS and Express API
  isolation tests for customer, barber, and owner roles using ephemeral users
  generated by the test suite. See
  `LOCAL-SUPABASE-VERIFICATION.md`.
- Shop-owner requests now use a full verification lock: `/verification` is the
  only reachable signed-in screen until approval, with Sign out as its only
  action. `Layout` and `RequireAuth` block direct-route bypasses, Express blocks
  operational API calls, and migration `20260717000700` repeats the rule for
  direct authenticated Postgres access. Pending barber hiring flow is unchanged.

### Appointment backend lifecycle (2026-07-18)

- `appointments` now uses a versioned canonical lifecycle: requested,
  confirmed, checked-in, in-progress, awaiting confirmation, completed,
  declined, expired, cancelled, customer no-show, or disputed.
- Express exposes explicit actor-scoped commands instead of arbitrary status
  writes. Owner: accept/decline/reassign/resolve. Assigned barber: issue
  check-in code/start/finish/no-show. Customer: check-in/confirm/dispute/cancel.
  All participants can read the immutable timeline.
- Each lifecycle mutation is a service-role-only PostgreSQL function with a
  locked row, optimistic `expected_version`, repeated authorization/state
  checks, and an `appointment_events` insert in the same transaction.
- A one-minute API worker expires unanswered requests and finalizes finished
  appointments after the customer confirmation window.
- Appointment service name/duration/price are snapshotted for historical
  reporting. Dashboard `revenue_cents` remains a compatibility estimate until
  actual payment records exist.
- Local Supabase integration proves RLS isolation, the owner check-in -> barber
  start/finish -> customer completion flow, stale-write conflicts, automatic
  transitions, event privacy, and owner reassignment.
- Migration `20260718000200` fixes SQL NULL handling that accidentally locked
  ordinary customers under the restrictive owner-verification policy.

## Data model additions (mock DB v15-v17)

New shared types (`packages/shared/src/types.ts`):
- `BarberEmployment` â€” one record per shop stint (`hired_at`, `ended_at`).
  `joinWithCode` closes the old stint and opens a new one â†’ per-shop
  attendance reset happens structurally.
- `BarberAbsence` â€” barber + shop + date (+ private reason).
- `ShiftChangeRequest` â€” barber + shop + date + message + status
  (`pending|approved|declined`).
- `Appointment.shop_id` â€” assigned when a booking is made or rescheduled so
  its historical shop does not change when its barber moves to another roster.

New service methods on `BarberEmploymentService`: `getMyEmployment`,
`listMyAbsences`, `listMyShiftChangeRequests`, `requestShiftChange` â€” all
scoped to the ACTIVE employment record.

New pure rule: `summarizeBarberAttendance` in
`packages/shared/src/attendance.ts` (present = scheduled day without an
absence record; returns month + tenure summaries).

v17 adds owner tools: `StaffNote` type + `staffNotes` collection, and new
contract methods â€” `bookings.listForMyShop`, `chat.openStaffConversation`,
`employment.listMyShopStaff` / `setBarberRules` /
`resolveShiftChangeRequest` / `addStaffNote` (lahat guarded ng
`requireOwnedShop()`). Owners also gained chat access to their shop's
threads (`canAccessShopConversation`).

Seed (`services/mock/seed.ts`): employment stints for all 11 roster barbers;
demo Miguel has an ~8-month tenure, 3 absences, and 1 pending shift request.
v17 seeds 5 listing-only customers + ~33 Tondo bookings sa nakaraang ~10
linggo (weighted si Mika Santos bilang top visitor) at 4 staff notes para
may totoong laman ang owner analytics. Absence/hire/booking dates are
generated relative to the real clock so the demo stays alive. Migration v15
upgrades employment data; v16 snapshots the shop on existing appointments;
v17 merges the owner-tools seed data.

## Known gaps / intentional mock limitations

- **Owner approval ng shift requests ay meron na** (Staff tab) â€” pero ang
  approval ay status change lang; hindi nito awtomatikong binabago ang shift
  (the owner edits the shift after approving, if needed).
- **Owner shift edits and barber self-service share one rules table** â€”
  parehong `availability rules` ang ine-edit ng `setBarberRules` (owner) at
  `setRules` (barber); walang conflict resolution beyond last-write-wins.
- **Attendance is derived** (schedule minus absences), not clock-in based.
- **Gear unlock thresholds are duplicated** between `DoodleAvatar.tsx`
  (UI catalogues) and `MockBackend.ts` (enforcement) â€” same precedent as
  `DOODLE_AVATAR_PATTERN`; comments on both sides point at each other.
- **Staff notes**: owner UI lang ang nakakasulat sa ngayon; the schema
  records `author_id` so barber-written notes are representable later.
- **Owner cannot start customer threads** â€” customers initiate; the owner
  replies (staff threads lang ang owner-initiated).
